## 一、项目背景

面向分校使用的印刷管理平台，用于管理教材的印刷需求。现有系统为单页 HTML 文件（v30），已实现产品 BOM 查询功能，连接 Teable 数据库。本次重构为 React 多页面应用，并新增合同管理、订单审核、分发管理、用户权限等完整功能。

---

## 二、技术栈

| 层 | 选型 |
|---|---|
| 构建工具 | Vite |
| UI 框架 | React 18 |
| 路由 | React Router v6 |
| 样式 | Tailwind CSS（超极简主义，UI skill 生成） |
| 状态管理 | React Context（Auth + 全局数据缓存） |
| 密码存储 | bcryptjs（客户端哈希，内部工具可接受） |
| Excel 导入/导出 | xlsx.js |
| HTTP | 原生 fetch（封装 Teable client） |
| 数据库 | Teable（yach-teable.zhiyinlou.com） |

**数据策略：** 所有数据一次性拉取全量，前端内存过滤，不依赖 Teable 服务端查询参数。

**安全说明（已知限制）：** 本系统为内部工具，采用纯前端架构。Teable 服务账号 token 打包在客户端产物中，前端角色权限仅做界面层隔离，无法阻止熟悉技术的用户绕过。密码采用 bcryptjs 客户端哈希后存储，原始密码不落库，但哈希值本身在此架构下具有等效凭证风险。这些是纯前端方案的已知折衷，适用于内部管理工具场景。

## 五、核心业务流程

### 5.1 认证流程

**登录：**
1. 输入用户名密码
2. 前端拉取用户表全量数据
3. 找到匹配用户名的记录，bcryptjs 比对密码哈希
4. 检查状态：待审核→提示「账号审核中，请联系校区管理员」，已驳回→提示「审核未通过」，已激活→继续
5. 解析角色表中该角色的权限配置 JSON，写入 AuthContext + localStorage
6. 根据权限渲染可访问的导航项，跳转首页

**注册：**
1. 用户填写：用户名（校验唯一性）、密码、姓名、所属分校、所属校区
2. 客户端 bcryptjs 哈希密码（cost=10），写入用户表，状态=「待审核」，角色ID=空
3. 拥有「approve_users=true」权限且「负责校区」包含该用户所属校区的用户，在用户管理页的「待审核」tab 看到新申请（页面进入时重新拉取用户数据，顶部导航显示待审核数量徽标提示）
4. 审核管理员通过 → 在弹窗中为该用户分配角色（必填）→ 状态改为「已激活」
5. 审核管理员驳回 → 状态改为「已驳回」

**角色为空的 fallback 处理：**
用户状态为「已激活」但角色ID为空时（异常情况），登录后权限配置默认全为 false，可访问首页看板，其余页面显示「无访问权限」提示，并提示联系管理员分配角色。

### 5.2 合同管理页面

合同管理页面展示两层信息：

**合同列表（上层）：** 从合同表读取，展示字段：合同编号、合同名称、有效期开始、有效期结束、适用分校、是否有效（前端计算今日是否在有效期内）。支持新增/编辑合同元信息（合同编号、合同名称、有效期、适用分校）。

**合同价格明细（下层/展开）：** 点击任一合同行，展开该合同的阶梯价格明细——从基础价格表中前端筛选「合同名称」匹配的所有记录，按类型/规格分组展示：数量区间（数量起～数量止）、印刷单价。支持在此处新增/编辑/删除该合同的价格条目（直接操作基础价格表对应记录）。

**合同名称一致性说明：** 合同名称作为跨合同表、基础价格表、订单主表的文本关联键。历史订单一经提交即保存合同名称快照，不受后续合同名称修改影响。编辑合同名称时前端显示警告提示，建议用新合同替代而非修改已有合同名称。

### 5.3 创建订单流程

**前置说明：**
- 首次创建订单（无现成 BOM）：用户手动填写规格，即产品名称、类型、成品尺寸、装订要求、封面/内页、纸张种类、纸张品牌、印刷要求、工艺要求、单BOM印刷数量，再填写印刷数量
- 基于 BOM 创建：从产品BOM表选择已有 BOM，自动填充上述规格字段

**流程：**
1. 选择方式：「从BOM新建」或「手动填写规格」
2. 填写/确认印刷规格及印刷数量
3. 选择主合同（下拉展示当前有效合同，过滤条件：有效期覆盖今日 且 适用分校包含当前用户所属分校）
4. 可选选择对比合同（可选所有合同，包含已过期合同）
5. 系统自动价格匹配（见 5.3.1）
6. 实时展示：主合同总价、对比合同总价、节约金额（对比合同总价 - 主合同总价）、节约率（节约金额 / 对比合同总价）
7. 提交 → 写入订单主表（状态=待审核）+ 写入订单明细表
8. 可选「同时保存为BOM」→ 将规格写入产品BOM表（去重：以「所属分校 + 产品名称 + 成品尺寸 + 装订要求 + 纸张种类 + 印刷要求」六个字段的组合判断是否重复，若已存在则提示「该规格BOM已存在」，不重复写入）

#### 5.3.1 价格匹配规则

从基础价格表中依次匹配以下全部字段：
- 合同名称 = 所选合同名称
- 所属分校 = 当前用户所属分校
- 类型 = 订单类型
- 成品尺寸 = 订单成品尺寸
- 装订要求 = 订单装订要求
- 封面/内页 = 订单封面/内页
- 纸张种类 = 订单纸张种类
- 纸张品牌 = 订单纸张品牌
- 印刷要求 = 订单印刷要求
- 工艺要求 = 订单工艺要求
- 数量起 ≤ 印刷数量 ≤ 数量止（取落入的数量区间）

**匹配异常处理：**
- 无匹配结果 → 前端显示「当前规格在所选合同中无对应价格」，阻止提交
- 多条匹配结果（数据异常）→ 取第一条并在界面提示「发现多条价格记录，已取第一条」

**总价计算：**
每条明细：`印刷总价 = 印刷数量 × 印刷单价`
订单总价 = 所有明细印刷总价之和

### 5.4 订单审核流程

1. 拥有「approve_orders=true」权限的用户进入查询订单页
2. 筛选「待审核」状态订单（分校用户仅可见本分校订单）
3. 点击订单查看详情（主表信息 + 明细列表）
4. 审核通过 → 更新订单主表：订单状态=「已审核」
5. 驳回 → 弹窗填写驳回原因 → 更新订单主表：订单状态=「已驳回」，驳回原因=填写内容

### 5.5 分发流程

**背景：** 订单由分校下单，分发是将订单的总印刷数量拆分给该分校下属的各校区。

1. 进入分发页，列出「已审核」状态的订单（当前用户所属分校）
2. 选择一个订单点击「开始分发」
3. 系统从校区表加载该分校下的所有校区列表（含地址/收件人/电话）
4. 分配方式（二选一）：
   - **手动分配：** 勾选参与配送的校区，在每个校区行输入分配数量；右侧实时显示已分配总量 vs 订单总量，已分配总量 ≠ 订单总量时禁用提交按钮并红色提示
   - **导入分配：** 上传 Excel 文件（模板格式：校区名称 | 分配数量），系统读取后展示预览，同样校验总量等于订单数量，验证通过才可提交
5. 提交分发 → 写入分发表（每个参与校区一条记录，共用同一个分发单号），状态=「待确认」；**同一订单可多次分发**（例如补单）：每次分发生成新的分发单号，分发表中同一订单编号可有多个分发单号对应的记录组，导出时按分发单号分组
6. 校区负责人（负责校区字段包含该校区）登录后，在分发页的「我的分发」tab 查看本人负责校区的分发记录
7. 校区负责人确认收货 → 更新分发表该条记录状态=「已确认」
8. **导出分发清单：** 点击导出按钮，系统关联校区表，生成 Excel 文件，列：订单编号、校区名称、所属分校、地址、收件人、电话、分配数量、状态
